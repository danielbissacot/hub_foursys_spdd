Fluxo SDD — Spec-Driven Development

O engine implementa uma metodologia rigorosa de 6 fases sequenciais. Cada fase gera artefatos no diretório doc_projeto/ que alimentam a próxima fase.

0 Constitution 1 Specify 2 Plan 3 Tasks 4 Implement 5-6 QA
0
Constitution — Governança
Modelo: Haiku (Light)

Estabelece os princípios arquiteturais, padrões de qualidade e regras de governança do projeto. É o "contrato" que todas as fases seguintes respeitam.

Comando Chat:
@foursys_sdd /constitution
Artefato gerado:
doc_projeto/constitution.md
Playbook:
[stack]/foursys-constitution.md
SOLID principles TDD requirements Golden rules AppSec guidelines
1
Specify — Refinamento do User Story
Modelo: Haiku (Mini)

Analisa e refina a User Story com base na metodologia INVEST. Gera critérios de aceite em BDD e detecta automaticamente a stack tecnológica.

Comando Chat:
@foursys_sdd /specify
Artefatos:
user_story.md
technical_spec.md
Playbook:
foursys-specify.md
foursys-specify-tech.md
INVEST validation BDD criteria Auto stack detect Compliance score 0-100%
Ao salvar user_story.md, o plugin abre automaticamente technical_spec.md em split view e sugere a análise.
2
Plan — Especificação Técnica
Modelo: Haiku (Light)

Gera a arquitetura técnica e estratégia de implementação com base nos artefatos das fases anteriores. Inclui diagramas de sequência, tabelas de decisão e patterns específicos da stack.

Entradas:
constitution.md
user_story.md
technical_spec.md
Artefato:
implementation_plan.md
Contexto:
Lê os 2 arquivos fonte mais recentes (max 80 linhas) para adaptar à estrutura do projeto
3
Tasks — Decomposição em Tarefas
Modelo: Haiku (Light)

Decompõe o plano em tarefas atômicas, divididas em 2 sessões de implementação para evitar sobrecarga do contexto AI.

Sessão 1 — Domínio (max 50% das tasks)
Entidades, interfaces, domain services, value objects
Sessão 2 — Infraestrutura
Controllers, adapters, configs, integrações externas
Estimativas por tamanho: XS S M L  ·  Análise de impacto sistêmico (arquivos globais afetados)  ·  Dependências entre tasks
4
Implement — Geração de Código
Modelo: Sonnet 4.6 (Implement)

Abre o Copilot Chat com um prompt pré-configurado referenciando os artefatos do doc_projeto/ e injetando a skill específica da stack ativa.

foursys.implement
Implementação completa
foursys.implementSession1
Apenas Domínio
foursys.implementSession2
Apenas Infraestrutura
Arquivos existentes são marcados como "protegidos" — o AI não os modifica sem task explícita.

Figma MCP — Design Integration Angular only

Integração com o servidor MCP oficial do Figma para importar designs diretamente como contexto do Copilot Chat, gerando critérios de aceite visuais automaticamente.

📦Estado 1 — Instalar

Extensão Figma não instalada. Botão aparece na sidebar.

Botão: 📦 Figma
Ação: Instala figma.figma-vscode-extension
Após: Recarrega janela + prompt de login no Figma
⚙️Estado 2 — Configurar MCP

Figma instalado, mas MCP não configurado.

Botão: ⚙️ Figma MCP
Ação: Escreve mcp.json automaticamente
Config: https://mcp.figma.com/mcp
🎨Estado 3 — Importar

Figma + MCP ativos. Botão ativado na sidebar.

Botão: 🎨 Figma MCP
Ação: Solicita URL do Figma → salva em figma_ref.txt → abre Chat com análise do design
O que o MCP analisa no Figma
  • Layout, cores e elementos visuais do frame
  • Componentes do Liquid Design System / DS Bradesco
  • Estados visíveis (loading, vazio, erro, sucesso, validação)
  • Gera critérios de aceite visuais para Angular (Material / PrimeNG)
  • Referencia o user_story.md como contexto

Override Manual

Para forçar um modelo específico em todas as fases:

{
  "foursys.modelOverride": "claude-sonnet-4-6"
}

Deixe vazio ("") para usar seleção automática por fase.

Configurações (settings.json)
{
  // Força família de modelo específica (vazio = auto-select por fase)
  "foursys.modelOverride": "",

  // Budget de tokens de entrada por request (0 = desabilitar aviso)
  "foursys.tokenBudget": 4500,

  // URL base do Jira para export ao Xray
  "foursys.xrayJiraUrl": "https://jira.sua-empresa.com.br",

  // Chave do projeto Jira
  "foursys.xrayProjectKey": "PROJ"
}
SettingTipoPadrãoDescrição
foursys.modelOverridestring""Override do modelo IA (família)
foursys.tokenBudgetnumber4500Limite de tokens de entrada por requisição
foursys.xrayJiraUrlstring""URL base do Jira (ex: https://jira.empresa.com)

Secrets (armazenados com segurança): foursys.xrayApiToken (token Bearer do Jira) e foursys.mendLicenseKey (licença Mend) são armazenados via VS Code Secrets API — nunca em texto plano. Configure via: Ctrl+Shift+P → Foursys: Configurar Token Xray

Integração Xray / Jira
Configuração
  1. Configure foursys.xrayJiraUrl e foursys.xrayProjectKey no settings.json
  2. Execute: Ctrl+Shift+P → Foursys: Configurar Token Xray
  3. Insira o Bearer token do Jira (armazenado com segurança)
Workflow de Export
  1. Execute QA-2: gera casos_teste.md em Gherkin (inglês)
  2. Execute QA-3: gera roteiros_teste.md
  3. Clique 📤 Exportar para Xray na sidebar QA
  4. O plugin valida keywords Gherkin (devem ser em inglês)
  5. Envia via API REST ao Jira Xray
  6. Arquivo local: qa/xray_export.feature
Atenção: Os casos de teste BDD devem ter keywords Gherkin em inglês (Given, When, Then, And, But) para compatibilidade com o Xray. O plugin valida e avisa se encontrar keywords em português.
Cenários de Uso

Constitution → Specify → Plan → Tasks → Implement → QA

Ciclo completo com rastreabilidade total via doc_projeto/. Ideal para features novas em projetos existentes ou projetos do zero.

Specify story → Plan incremento → Tasks → Session 1 → Session 2

Adicionar features a um sistema existente com contexto de workspace injetado automaticamente.

Constitution COBOL → Specify modernização → Plan transpilação → Implement → QA

Usar skills especializadas: cobol-reversa-completa, cobol-transpilacao-java, cobol-analise-abend.

Constitution AppSec → Plan → Implement → Mend Scan → QA Report

Foco em compliance e segurança, com scan de CVEs integrado e relatório final de qualidade.
